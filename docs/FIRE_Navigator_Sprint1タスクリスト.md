# FIRE Navigator Sprint1 実行タスクリスト

位置づけ: 「MVP定義 & Sprintロードマップ」のSprint1（基盤構築）を、実際に手を動かせる作業単位まで分解したもの。Sprint1の範囲・終了条件（「Excelの特定セルに数値を入れてランチャーを実行すると、Pythonがその値を読み込み、加工してExcelに書き戻す」）は変更しない。

---

## 0. 開発環境の取り決め（Sprint1に入る前に決めておくこと）

### 0.1 Pythonバージョン

**推奨: Python 3.12以上**

- 型ヒント関連の機能（`Protocol`、`dataclass`の改善等）が安定しており、Domain Layerの値オブジェクト設計（Money/Rate等）と相性が良い。
- 特別な理由がなければ、手元の環境で使える最新の安定版を使えばよい。バージョンを固定する目的は「将来の互換性トラブルを防ぐこと」であり、最新性そのものを追う必要はない。

### 0.2 依存関係・仮想環境管理

2つの選択肢を比較する。

| 方式 | メリット | デメリット |
|---|---|---|
| **venv + requirements.txt** | 標準ライブラリのみで完結し、追加ツールの学習コストがゼロ。構成がシンプルで、これまでの「シンプルで安全性の高い方法を選ぶ」という方針と一貫する。 | 依存関係のバージョン固定（lock）がrequirements.txtの手動管理に頼る。大規模プロジェクトではやや心もとない。 |
| **Poetry** | 依存関係の解決とlockファイルによる再現性が高く、「5年後・10年後でも保守できる」という長期保守性の思想には合致する。 | ツール自体の学習コスト・セットアップの手間が増える。 |

**推奨: venv + requirements.txt を採用する。**
理由: FIRE Navigatorは自分専用の個人プロジェクトであり、チーム開発のような依存関係の厳密な再現性よりも、環境構築のシンプルさ・トラブルの少なさを優先すべき局面である（Excel起動方式の判断と同じ考え方）。将来、依存パッケージが増えて管理が煩雑になった時点でPoetryへの移行を検討すればよい（YAGNI）。

**Sprint1でやること**:
- [ ] `python -m venv .venv` で仮想環境を作成
- [ ] `requirements.txt` を作成（最初は空、またはpytest等の最低限のみ）
- [ ] `.gitignore` に `.venv/` を追加

## 2. ランチャー方式の技術検証

「v1.1採用ロードマップと実行方式」で決めた**外部スクリプト手動実行方式**を、実際に動かして確認する。

### 2.1 最小の動作確認（Hello World相当）

- [ ] `scripts/run_sample_simulation.py` に、まずは「実行されたらコンソールに文字を出すだけ」の最小コードを書く

- [ ] ランチャーを「どのフォルダから実行しても」正しく動くように、スクリプトのパスをランチャー自身の場所からの相対パスで解決する（実行時のカレントディレクトリに依存させない）
- [ ] 入力Excelファイルの場所（例：プロジェクト直下の`data/input.xlsx`）を固定の相対パスとして決めておく

## 3. Excel⇄Python 最小疎通の実装

MVP定義のSprint1終了条件そのもの。

- [ ] `openpyxl`を`requirements.txt`に追加し、`pip install -r requirements.txt`で導入
- [ ] `data/input_sample.xlsx` を作成し、適当なシートに数値を1つ入れておく（例：`A1`セルに`100`）
- [ ] `adapters/excel/excel_input_adapter.py` の最小版を実装：指定セルの値を読み込む関数
- [ ] `scripts/run_sample_simulation.py` を拡張：読み込んだ値を2倍にする（Sprint1時点では計算ロジックではなく疎通確認が目的なので、ダミー処理でよい）
- [ ] `adapters/excel/excel_output_adapter.py` の最小版を実装：計算結果を指定セルへ書き込む関数
- [ ] ランチャーを実行し、「入力Excelの値を変更 → ランチャー実行 → 出力Excel（または同ファイルの別セル）に2倍の値が反映される」ことを確認する

この時点でSprint1の終了条件（「動くもの」）が満たされる。

## 4. テスト基盤の雛形

- [ ] `pytest`を`requirements.txt`に追加
- [ ] `tests/unit/`, `tests/integration/`, `tests/regression/`, `tests/fixtures/` の空フォルダを作成（`tests/regression/golden/`も含む）
- [ ] 3章で作ったExcel Adapterの最小関数に対して、`tests/integration/`に1件だけ簡単なテストを書く（サンプルExcelを読み込んで期待通りの値が取れるか）
- [ ] `pytest`コマンドでこのテストが通ることを確認する

CIの自動実行（GitHub Actions等）は個人開発の初期段階では必須ではない。手元で`pytest`が通ることをまず確認できれば十分であり、CI導入はSprint2以降、必要性を感じたタイミングで検討すればよい（これもYAGNIの考え方に沿う）。
