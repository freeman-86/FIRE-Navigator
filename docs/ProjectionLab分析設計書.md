# ProjectionLab 分析設計書

対象: ProjectionLab（financial planning / FIREシミュレーションSaaS, 開発元: Kyle Nolan氏）
作成目的: FIRE Navigator開発における競合分析・設計リファレンス
注記: 本書は公式サイト・ヘルプセンター・レビュー記事・リリースノート等の公開情報から作成した外部分析であり、内部のデータベース設計・ソースコードは非公開のため、UI仕様から論理的に推測した箇所を含む。

```
[Welcome]

        │
        ▼
   [About You] → [保有資産の選択] → [残高入力（カテゴリ反復）]
        │                                   │
        │                                   ▼
        │                          [Investments詳細入力]
        │                                   │
        ▼                                   ▼
  [Real Assets入力] ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ┘
        │
        ▼
  [税設定（概算）]
        │
        ▼
  [初期アウトプット（ネットワース速報グラフ）]
        │
        ▼
┌───────────────────────────────────────────┐
│            Dashboard（常設ホーム）           │
│  ここから各機能タブへ自由に遷移（非直線的）    │
└───────────────────────────────────────────┘
        │
   ┌────┼─────────────┬─────────────┬─────────────┐
   ▼    ▼             ▼             ▼             ▼
[Current Finances] [Plan編集群]  [Advanced群]   [Compare]
   │                 │             │
   ▼                 ▼             ▼
[Progress自動記録] [Milestones/  [Monte Carlo/
                    Income/       Withdrawal/
                    Expenses/     Cash Flow(Sankey)/
                    Cash Flow     Tax Analytics/
                    Priorities/   Roth Conversion/
                    Real Assets/  Estate]
                    Debt/Tax]
```

**遷移の設計思想**:
- 初回オンボーディングのみ**ウィザード型（直線的・強制順序）**。ステップごとに次に何を聞かれるか予測可能にし、離脱を防ぐ。
- オンボーディング完了後は**ハブ&スポーク型**に切り替わる。Dashboardを中心に、どのタブにも1クリックで行き来できる非モーダル構造。
- 各入力パネルはダッシュボードから**オーバーレイ／サイドパネルで開閉**する設計と見られ、画面全体の遷移（ページ replace）ではなく、グラフを表示したまま設定を差し替える「即時反映」型（後述UXの特徴）。
- アプリ上部のバーからマイルストーンや成長率・インフレ率をスライダーで直接編集でき、"編集→グラフ再描画"のループが最短で完結する。

## 3. 入力項目

### 3.1 プロフィール／基本情報
- 生年月日（本人・配偶者）
- 居住国・州／地域（税制プリセットの決定要因）
- 配偶者の有無、扶養家族

### 3.2 資産（Accounts）
- 現金・普通預金
- 課税投資口座
- 退職口座（確定拠出型：401k, IRA等／確定給付型年金）
- HSA（医療貯蓄口座）
- 529（教育貯蓄プラン）
- 暗号資産
- 各口座について：残高、成長率仮定、本人/配偶者/共同名義（共同名義は50/50で按分）

### 3.3 負債（Debt）
- 種別（学生ローン、住宅ローン、その他無担保債務等）
- 残高、金利、返済スケジュール、繰上返済設定

### 3.4 実物資産（Real Assets）
- 住宅、車、家具、ボート、貴金属など
- 取得予定時期、価格、将来の売却/買替えイベント、資産固有の費用仮定（維持費等）

### 3.5 収入・キャリア（Income）
- 給与、昇給率、キャリア進行（転職・昇進イベント）
- 退職後の収入（年金、配当等）

### 3.6 支出（Expenses）
- 生活費の各カテゴリ（住宅ローンは資産側で計上するため対象外）
- 支出の時系列変化（増減イベント）
- Flex Spending（柔軟支出：市場状況に応じ増減させる支出）

### 3.7 マイルストーン（Milestones）
- 退職年齢、プラン終了年齢（寿命）
- FI達成条件（例：純資産が年間支出の25倍＝4%ルール）
- 転居（州・国の変更）
- 任意の追加マイルストーン（複数条件によるトリガーも設定可）

### 3.8 キャッシュフロー優先順位（Cash Flow Priorities）
- 緊急資金の目標額
- 投資への追加拠出ルール
- 負債への追加返済ルール
- 口座間振替（Transfers）、外部への贈与・寄付（Gifts, Donor-Advised Fund拠出）

### 3.9 税設定（Tax）
- 所得税・キャピタルゲイン税率（自動推定 or カスタム設定）
- 国別プリセット（米国、英国、豪州、ドイツ、スペイン等）
- Qualified Charitable Distribution（QCD）、Donor-Advised Fund（DAF）等の税戦略設定

### 3.10 市場前提（Assumptions）
- インフレ率
- 投資成長率（株式/債券配分別）
- 配当利回り
- 資産配分（株式/債券比率）の時系列変化

### 3.11 取り崩し戦略（Withdrawal Strategy）
- 口座取り崩し順序（現金→課税→非課税 等のカスタム順）

## 4. データ構造（推定）

内部実装は非公開だが、UI・ヘルプ文書から次のような論理データモデルが推測される。

```
User
 ├─ Profile { 生年月日, 配偶者情報, 居住地, 通貨設定 }
 ├─ Plan[]  ← ユーザーは複数プランを保持可能
 │    ├─ PlanMeta { 名前, 開始条件(Today基準 or Fixed Date), 開始日 }
 │    ├─ Assumptions { インフレ率, 成長率, 配当利回り, 株式/債券配分の時系列 }
 │    ├─ Milestone[] { 種別, トリガー条件(年齢/日付/複合条件), パラメータ }
 │    ├─ Account[]  { 種別(現金/課税/退職/HSA/529/暗号資産),
 │    │               所有者(本人/配偶者/共同),
 │    │               残高, 成長率, 拠出設定 }
 │    ├─ Debt[]      { 種別, 残高, 金利, 返済条件 }
 │    ├─ RealAsset[] { 種別, 取得/売却イベント, 価格, 関連費用仮定 }
 │    ├─ IncomeStream[] { 種別, 金額, 成長率, 開始/終了条件 }
 │    ├─ Expense[]   { カテゴリ, 金額, 時系列変化イベント, Flex設定 }
 │    ├─ CashFlowPriority[] { 順位, 種別(貯蓄/投資/返済/振替/贈与), 条件 }
 │    ├─ TaxConfig   { 地域プリセット or カスタム税率, 特殊戦略設定(QCD/DAF等) }
 │    ├─ WithdrawalStrategy { 口座取り崩し順序 }
 │    └─ SimulationResult（キャッシュ） 
 │         ├─ YearlyProjection[] { 年, 各口座残高, 収入, 支出, 税額, ネットワース }
 │         ├─ MonteCarloResult { 試行回数, 成功確率, パーセンタイル分布 }
 │         ├─ TaxAnalytics { 年別実効/限界税率, 所得種別ごとの内訳 }
 │         └─ EstateResult { 総遺産, 相続税, 純遺産(Net Legacy) }
 └─ ProgressLog[] ← Current Financesの更新履歴（実績スナップショット）
      { 記録日時, 各口座残高スナップショット }
```

**設計上の特徴（推定）**:
- 「入力データ（Plan設定）」と「シミュレーション結果（Result）」を明確に分離。入力を変更すると結果はキャッシュ再計算される構造と見られる（スライダー操作での即時再描画から推測）。
- `ProgressLog`（実績）と`Plan`（将来予測）を分離することで、「予測 vs 実績」の比較を可能にしている。
- マイルストーンが「複数条件のトリガー」を持てる＝条件分岐を表現できるツリー構造（"multi-condition milestones"でシナリオ分岐を作れるとユーザーコメントにあり）。
- 通貨・時点表現として「Today's Currency（実質値）」と「Actual（名目値）」の二重表現を持つ（固定収入の実質/名目切替のヘルプ項目より）。

## 5. シミュレーション項目

| 項目 | 内容 |
|---|---|
| 決定論的シミュレーション（Deterministic Projections） | 単一の成長率・インフレ率仮定に基づく将来予測（基本モード、無料でも利用可） |
| ヒストリカル・バックテスト（Historical Backtesting） | 過去の実際の市場データ（複数年度）を当てはめて資産推移を再現 |
| モンテカルロ・シミュレーション | 過去の市場のボラティリティに基づき多数の試行を行い、プランの「成功確率（Chance of Success）」をパーセンタイル分布で提示。ミルストーン単位でのタイミング/結果によるフィルタも可能 |
| キャッシュフロー・シミュレーション | 収入→支出→貯蓄/投資/返済という年次資金フローの計算（優先順位ルールに基づく） |
| 税シミュレーション（Tax Estimation） | 地域別税制に基づく所得税・キャピタルゲイン税・実効/限界税率の推定 |
| Roth変換シミュレーション | Roth変換タイミング・金額を変えた際の生涯税負担・遺産への影響分析 |
| 取り崩し戦略シミュレーション | 口座取り崩し順序の違いによる資産寿命・税負担の比較 |
| 相続・遺産シミュレーション（Estate） | 相続税（米国連邦、英国相続税等）を考慮した純遺産（Net Legacy）計算 |
| シナリオ比較（Compare） | 複数プランを並べて主要指標を比較 |
| 感度分析（スライダー操作） | 成長率・インフレ率などをスライダーで動かし、結果への感応度をリアルタイム確認 |

## 8. UXで優れている点

1. **即時フィードバックループ**: 上部バーのスライダーで成長率・インフレ率を動かすと、メイングラフがリアルタイムで再描画される。「入力→出力」の往復が最短で、感覚的に感度分析ができる。
2. **ハブ&スポーク型の常設ダッシュボード**: 一度セットアップが終われば、どの設定画面にも1クリックで行き来できる。深い階層に潜らせない設計。
3. **ウィザード型オンボーディングと自由編集の使い分け**: 初回は迷わせない直線的フロー、以降は自由な非線形操作。学習コストと自由度のバランスが取れている。
4. **視覚的な資金フロー表現（Sankey図）**: 数字の羅列ではなく「お金がどこから来てどこへ流れるか」を直感的に理解させる。年単位でスライドして確認できる操作性も高評価。
5. **ビジュアルデザインの一貫性と質**: 競合（Excelベースのツールや従来型電卓アプリ）と比較して、モダンで読みやすいグラフ、ダークモード対応など「使っていて楽しい」体験を重視している点がレビューで繰り返し評価されている。
6. **「予測」と「実績」の分離と接続**: Current Finances（実績更新）とProgress（実績記録）を分けつつ、当初計画との比較を可能にしており、長期利用者の継続的なモチベーションになっている。
7. **段階的な機能開放（フリーミアム設計）**: 無料でも基本的な決定論的予測・モンテカルロ・カスタムプロットまで試せるため、価値を体験してから課金する導線が機能している。
8. **複合条件マイルストーン**: 単純な日付/年齢だけでなく、複数条件によるシナリオ分岐を組めるため、「もし市場が悪化したら」等の条件付き意思決定を表現できる（ユーザーの評価コメントでも支持されている）。
9. **国際化対応**: 米国以外の税制（英国、豪州、ドイツ、スペイン等）にも対応しており、単一市場前提のツールが多い中で差別化要因になっている。

## 9. 改善できそうな点

ユーザーコミュニティ（ユーザーの要望一覧等）や第三者レビューから読み取れる、改善余地・弱点。

1. **無料版でのデータ非保存**: 無料（Basic）プランはセッション間でデータが保存されず、本格的に使うには課金が前提になる。「アカウント作成前にデータ保存の有無を明示してほしい」という否定的な声がある。
2. **非米国ユーザーへの税制対応の粒度**: 英国の確定給付年金（DBペンション）や豪州のスーパーアニュエーションなど、国ごとの制度の細部（特殊な非課税枠、累進計算の癖）を再現するには手動調整が必要で、直感的に設定しづらいという声がある。
3. **入力パネルとグラフの同時参照性**: 「Accounts・Income・Expenses・Real Assets・Cash Flow PriorstiesをPlanグラフの隣に並べて表示したい」という要望があり、現状は入力パネルとグラフを行き来する必要がある（同一画面での並列参照ができない）。
4. **入力パネルのウィンドウサイズ調整不可**: 各設定パネル（Accounts, Income, Expense等）の表示領域サイズを変更できず、項目数が多い場合に閲覧性が下がるとの指摘がある。
5. **特殊な収入形態への対応の弱さ**: 409A（繰延報酬）プランなど、一部の複雑な税繰延スキームに対するネイティブ対応が手薄との要望がある。
6. **一括編集機能の不足**: 複数項目のコピー等、まとめて操作する手段が限られており、類似項目を多数作る際の手間が大きい。
7. **サマリー機能の物足りなさ**: 「カスタマイズ可能な純資産・支出のブラケットに基づく退職年齢のサマリー表」など、複数プランを横断した俯瞰的な要約機能が求められている。
8. **プラン変更の影響範囲の制御**: 「What-if」的な変更が、本来変わってほしくない過去期間（現在日以前）の投影線にまで影響してしまうケースがあり、変更の適用範囲（未来のみ／全期間）を明示的に選べる設計が望まれている。
9. **グラフのY軸自動スケーリング**: 表示範囲に応じてY軸が自動調整されない場面があり、比較や読み取りがしづらいとの指摘がある。
10. **モバイル体験は「対応はしているが本命ではない」**: 公式ヘルプでも「基本的にはPCでの利用を推奨」と明言されており、複雑な入力を伴うツールとしてモバイルUXは補助的位置づけに留まっている（FIRE Navigatorのように最初からマルチデバイスを想定するなら差別化点になり得る）。

---

## 総括（FIRE Navigator設計への示唆）

- **役割分担の観点**: ProjectionLabは「入力UI」と「計算エンジン」が一体のSaaSだが、FIRE Navigatorの「Excel＝入力/レポートUI、Python＝エンジン」という分離方針は、ProjectionLabにない「オフライン・自己所有データ・カスタマイズ自由度」という強みになり得る。
- **データモデルの観点**: Plan / Assumptions / Milestone / Account / Debt / RealAsset / Income / Expense / CashFlowPriority / TaxConfig / WithdrawalStrategy / SimulationResult という分離構造は、そのままFIRE NavigatorのPythonエンジン側のドメインモデル設計の参考になる。
- **UXの観点**: 「即時フィードバック（スライダー的操作）」と「予測 vs 実績の分離・比較」は、Excel UIでも再現価値が高い設計思想。
- **改善点の観点**: ProjectionLabで指摘されている「入力パネルとグラフの同時参照性の低さ」「変更適用範囲の制御」は、Excelを主UIとするFIRE Navigatorであれば構造的に解決しやすい弱点であり、差別化ポイントとして意識する価値がある。
