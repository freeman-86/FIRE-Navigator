# FIRE Navigator MVP定義 & 開発ロードマップ（Sprint1〜10）

前提: 本書は「FIRE Navigator 機能方針書」（確定版）を判断基準とする。以後の新機能追加は、この方針書とMoSCoW分類に照らして是非を判断すること。

---

## 1. MoSCoW分類

方針書の「採用する機能」「日本仕様へ変更する機能」「追加する独自機能」に挙がった項目を、MVP（Sprint10終了時点）の観点で優先度づけする。「採用しない機能」は方針書の時点で対象外のため、本表には含めない（=最初からWon't Everとして扱う）。

### Must（MVPの根幹。これがなければ製品として成立しない）

| 機能 | 分類元 | 理由 |
|---|---|---|
| 決定論的シミュレーション（基本モード） | 採用 | 全機能の計算基盤。これがなければ他の何も動かない。 |
| ネットワース推移グラフ | 採用 | 「計算結果を見る」という最小の価値提供。 |
| Excel入力UIとの連携（読込／書出） | 独自追加 | Excel＝UI、Python＝エンジンという役割分担の根幹。これがないとツールとして体を成さない。 |
| 日本の税制コア計算（所得税・住民税・社会保険料・基礎控除等） | 日本化 | 手取りベースで計算できなければ、資産形成シミュレーションとして精度が出ず存在価値がない。 |
| NISA・iDeCo等の口座タイプモデル | 日本化 | 自分の実際の資産構成を表現するための最低限のデータモデル。 |
| キャッシュフロー優先順位（基本ルール） | 採用 | 「収入-支出の余剰をどう配分するか」がないと資産推移が計算できない。 |
| Git管理・シナリオ回帰テスト基盤 | 独自追加 | 商用レベルの品質・保守性という開発方針そのもの。後回しにすると技術的負債化するため最初から組み込む。 |

### Should（MVPに近い段階で欲しい。初回リリースの価値を大きく左右する）

| 機能 | 分類元 | 理由 |
|---|---|---|
| マイルストーン機能（基本トリガーのみ、複合条件は除く） | 採用 | 「FI達成」「退職」等の目標到達判定がないと意思決定支援として弱い。ただし複合条件は高度化なのでCoveへ。 |
| シナリオ比較（Compare） | 採用 | FIRE Navigatorの主目的（意思決定比較）に直結。Mustにしたいところだが、単一シナリオの精度が先。 |
| 感度分析（前提条件を変えて再計算） | 採用 | Excelとの親和性が高く実装コストは比較的小さいが、まずは単一計算が安定してから。 |
| 取り崩し順序（Withdrawal Strategy） | 採用 | 退職後シミュレーションに必須だが、退職前フェーズの精度確立が先。 |
| 公的年金（繰上げ/繰下げ）シミュレーション | 日本化 | 退職後資産寿命に直結する重要変数。ただし単独機能として後段で追加可能。 |
| カスタムプロット | 採用 | 見たい指標を選ぶ機能。基本グラフが安定してからでよい。 |

### Could（あれば価値は高いが、MVPの成立要件ではない。後回し可）

| 機能 | 分類元 | 理由 |
|---|---|---|
| モンテカルロ／ヒストリカルバックテスト | 採用 | 「確率で安心感を持つ」機能だが、決定論的シミュレーションが正確であることが前提であり優先度は一段落ちる。 |
| 予測vs実績（Progress） | 採用 | 継続利用の価値だが、そもそも「予測」が固まってから意味を持つ機能。 |
| Sankey資金フロー可視化 | 採用 | 理解を助けるが、数値・グラフだけでも意思決定は可能。UI実装コストも大きい。 |
| マイルストーンの複合条件トリガー | 採用（高度化） | 基本トリガーで最初は十分。複雑な条件分岐は後回し。 |
| iDeCo受給方法比較・退職所得控除最適受取 | 日本化／独自追加 | 精緻だが、退職が近づいてから使う機能であり緊急性が低い。 |
| 相続税（Estate） | 日本化 | 現時点のFIRE計画の主眼（資産形成・取り崩し）から見ると優先度は低い。 |
| 住宅税制の詳細（住宅ローン控除等） | 日本化 | 該当資産がある場合のみ必要。汎用モデルで代替可能な期間は後回し。 |
| ふるさと納税シミュレーション | 独自追加 | 便利だが独立性が高く、他機能への依存が少ないため後段でいつでも追加できる。 |
| NISA/iDeCo枠の使い切り最適化ロジック | 独自追加 | 基本の口座モデルが安定してからの最適化レイヤー。 |
| 為替リスク感応度分析 | 独自追加 | 海外資産保有時のみ必要。円建て一本化がMustなので、これは応用機能。 |
| 家計簿サービス連携（自動更新） | 独自追加 | 外部データ取込は仕様変動リスクが高く、手動更新（Mustの範囲）で当面代替可能。 |

### Won't（今回のスコープでは実装しない）

| 機能 | 理由 |
|---|---|
| 方針書「不採用」に分類した全機能（Advisor Dashboard、課金導線、フリーミアム設計、Discord等、Plugin API、多国籍税制対応、米国固有制度直接実装） | 方針書で明確に不採用と決定済み。再検討しない。 |
| 多通貨・海外送金等の本格的な国際対応インフラ | 為替リスク感応度分析（Could）で必要十分。本格インフラ化は自分専用ツールに過剰。 |

## 2. MVPの定義（Sprint10終了時点のゴール）

> **「Excelに自分の資産・収入・支出・NISA/iDeCo口座を入力すると、日本の税制・社会保険料を反映した手取りベースの資産推移が計算され、ネットワースグラフとして出力される。退職年齢や支出方針などの複数シナリオを比較でき、感応度分析・取り崩し順序・年金受給タイミングも扱える。すべての計算は回帰テストで保護され、変更履歴はGitで追跡される。」**

Must + Should を全て満たした状態をMVP完成とする。Could以降はMVP後のバックログとして方針書の優先順位に従い着手する。

## 3. 依存関係マップ（サマリ）

```
[基盤: Excel⇄Python I/O, Git/テスト基盤]
        │
        ▼
[ドメインモデル（Plan/Account/Income/Expense等）]
        │
        ▼
[決定論的シミュレーションエンジン（税抜き最小版）]
        │
        ▼
[ネットワースグラフ出力] ← ここでMVPの「核」が一周する
        │
        ▼
[日本税制コア（所得税/住民税/社会保険料）]
        │
        ▼
[NISA/iDeCo口座モデル + キャッシュフロー優先順位]
        │
        ├──────────────┐
        ▼              ▼
[マイルストーン基本]  [シナリオ比較]
        │              │
        └──────┬───────┘
               ▼
[感応度分析 + 取り崩し順序 + 公的年金シミュレーション]
               │
               ▼
[モンテカルロ/バックテスト + Progress実績比較]
               │
               ▼
[品質固め・回帰テスト強化・レポート整備・Could着手]
```

依存関係の要点:
- **Excel I/Oとテスト基盤が最初**：これがなければ何を作っても検証・接続ができない。
- **税制計算は「口座モデル」より先**：NISA非課税判定は所得税計算ロジックに依存するため、税制コアが先。
- **マイルストーン／比較は、税引後の資産推移が安定してから**：不正確な数値の上に意思決定支援機能を作っても意味がない。
- **モンテカルロ・Progressは最後の応用層**：決定論的な計算が信頼できて初めて、確率的検証や実績比較が意味を持つ。

## 4. Sprintロードマップ（Sprint1〜10）

各Sprintは「動くプロダクト」を終了条件とする。単なるコード片の完成ではなく、**Excelから実行して結果が得られる状態**を必須のゴールとする。

### Sprint 1：基盤構築
- **やること**: プロジェクト構造の確定（リポジトリ構成、モジュール分割方針）、Excel⇄Python間の最小I/O疎通（1シート・数セルの読込／書込）、CI・テストフレームワークの雛形、Gitブランチ運用ルールの確立
- **Sprint終了時に動くもの**: Excelの特定セルに数値を入れてマクロ/スクリプトを実行すると、Pythonがその値を読み込み、簡単な加工（例：2倍にする）をしてExcelに書き戻す
- **依存**: なし（最初のSprint）

### Sprint 2：ドメインモデル確定
- **やること**: Plan / Account / Income / Expense / Debt / Milestone 等のPythonデータクラスを設計・実装。Excelシート項目とのマッピング仕様書を作成
- **Sprint終了時に動くもの**: Excelの基本入力シート（資産一覧・収入・支出）を読み込み、Python内部でドメインオブジェクトに変換し、内容をコンソールまたはExcelの確認用シートに書き戻して「正しく読み取れていること」を確認できる
- **依存**: Sprint1のI/O基盤

### Sprint 3：決定論的シミュレーションエンジン（税抜き最小版）
- **やること**: 単年の「収入−支出＝余剰、資産×成長率」という最も単純なロジックで、複数年のネットワース推移を計算するエンジンを実装。最初の回帰テストケースを作成
- **Sprint終了時に動くもの**: Excel入力（資産・収入・支出・成長率）→Python計算→「西暦年別のネットワース数値テーブル」がExcelに書き戻される
- **依存**: Sprint2のドメインモデル

### Sprint 4：ネットワースグラフ出力
- **やること**: Sprint3の計算結果をExcelのグラフ機能（またはPython側でのグラフ生成→画像/オブジェクト埋め込み）として可視化。資産種別ごとの内訳（現金/投資等）も積み上げ表示
- **Sprint終了時に動くもの**: Excelの入力値を変更して実行すると、ネットワース推移グラフが自動更新される。**ここでMVPの核となる「入力→計算→可視化」の一周が完成する**
- **依存**: Sprint3の計算エンジン

### Sprint 5：日本税制コア実装
- **やること**: 所得税（累進課税）、住民税、社会保険料（健康保険・厚生年金・雇用保険）、給与所得控除・基礎控除・配偶者控除等をエンジンに組み込み、キャッシュフロー計算を「額面」から「手取り」ベースに変更
- **Sprint終了時に動くもの**: Excel入力（額面年収等）から手取り額が正しく計算され、税引後ベースのネットワース推移グラフが得られる
- **依存**: Sprint3の計算エンジン（税抜きロジックの上に税計算を組み込むため）

### Sprint 6：NISA/iDeCo口座モデル ＋ キャッシュフロー優先順位
- **やること**: 口座タイプ（NISA成長投資枠／つみたて投資枠、iDeCo、企業型DC、財形貯蓄、課税口座）ごとの非課税処理・拠出上限をモデル化。「緊急資金→NISA→iDeCo→課税口座」等の優先順位ルールに基づく自動配分ロジックを実装
- **Sprint終了時に動くもの**: NISA・iDeCoを使った場合と使わない場合で、税引後資産推移がどれだけ変わるかを比較できる（実運用可能な最小版として使い始められる状態）
- **依存**: Sprint5の税制コア（非課税判定は所得税計算に依存）

### Sprint 7：マイルストーン機能（基本）＋ シナリオ比較（Compare）
- **やること**: 退職年齢・FI達成条件（純資産が年間支出のN倍等）を「イベント」として定義し到達判定するロジックを実装。複数のプラン設定（シナリオ）を並列に計算し、主要指標を比較レポートとして出力
- **Sprint終了時に動くもの**: 「60歳退職」「65歳退職」など複数シナリオをExcel上で切り替えて計算し、結果を並べて比較できる
- **依存**: Sprint6（正確な税引後・口座別資産推移が前提）

### Sprint 8：感応度分析 ＋ 取り崩し順序 ＋ 公的年金シミュレーション
- **やること**: 成長率・インフレ率等の前提条件を複数パターンで自動的に振って再計算するバッチ処理を実装。退職後の口座取り崩し順序ロジックを実装。国民年金・厚生年金の繰上げ/繰下げ受給による受給額増減シミュレーションを追加
- **Sprint終了時に動くもの**: 退職後フェーズを含めた資産推移が、年金受給・取り崩し順序を考慮して正しく計算され、前提条件を変えた際の感応度がレポートとして得られる
- **依存**: Sprint7（マイルストーン＝退職イベントが前提として必要）

### Sprint 9：モンテカルロ／バックテスト（日本市場データ）＋ Progress実績比較
- **やること**: 日本株・全世界株・国内外債券の過去データセットを用いたモンテカルロシミュレーション（成功確率算出）を実装。実際の口座残高を記録し、当初プランとの比較（予測vs実績）を可視化する機能を実装
- **Sprint終了時に動くもの**: プランの「成功確率」がパーセンタイル分布として得られ、また実績データを入力すると計画線との乖離が確認できる
- **依存**: Sprint8（退職後を含む正確な決定論的計算が前提。確率的検証はその上に乗る応用層）

### Sprint 10：品質固め・回帰テスト強化・MVPリリース
- **やること**: Sprint1〜9で作成した全機能に対する回帰テストスイートを拡充し、カバレッジを確認。Excel⇄Python連携のエラーハンドリング・ドキュメント整備。Could項目のうち優先度が高いもの（例：カスタムプロット、または基本的なSankey可視化のいずれか一つ）に着手する余力があれば着手
- **Sprint終了時に動くもの**: Must＋Should全項目が揃い、回帰テストで保護された「MVP完成版」。以降はCould項目をバックログから優先順位に従って取り込むフェーズに移行する
- **依存**: Sprint1〜9すべて

## 6. 運用ルール（今後の追加機能について）

- 新しい機能案が出た場合は、まず「FIRE Navigator 機能方針書」の採用/不採用/日本化/独自追加のいずれに該当するかを確認する。
- 該当した上で、本書のMoSCoW分類（Must/Should/Could/Won't）に追加候補として位置づけ、Sprint10以降のバックログとして優先順位づけする。
- Mustの追加・変更は、既存Sprintの依存関係マップを見直した上でのみ行う（後工程への手戻りを避けるため）。
