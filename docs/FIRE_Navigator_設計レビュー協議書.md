# FIRE Navigator 設計レビュー協議書

版: v1.0（確定）

## 1. この文書の目的

本協議書は、FIRE Navigatorの設計品質向上を目的として、Claudeとの設計レビューを行うためのガイドラインである。

目的は「設計改善」であり、「仕様変更」ではない。

---

## 2. 前提

以下の文書は正式仕様である。

- FIRE Navigator機能方針書
- MVP定義・Sprintロードマップ
- システムアーキテクチャ設計書 v1.0
- システムアーキテクチャ設計書 v1.1
- v1.1採用ロードマップ
- Google Sheets移行メモ

これらの内容はレビュー対象ではあるが、レビューの目的は「理解した上で改善提案すること」であり、前提を無視して新しい設計へ置き換えることではない。

---

## 3. 絶対に変更してはいけない設計思想

以下はプロジェクトの憲法であり、変更提案の対象外とする。

- Engine First
- Source of Truth
- Google SheetsはView
- Python Engineのみが計算する
- DomainはUIを知らない
- JSONを唯一の共通インターフェースとする
- ライフプランニングエンジンという思想
- Decision Support
- Explainability
- Traceability
- 長期保守性

これらを変更する提案は採用しない。

## 6. 提案フォーマット・ランク

提案ごとに、以下をすべて記載すること（項目の重複を避けるため、書式はこの1つに一本化する）。

| 項目 | 内容 |
|---|---|
| 問題点 | 何が問題か |
| 根本原因 | なぜその問題が起きているか |
| 改善案 | どう直すか（理由・メリット・デメリット・実装コストを含む） |
| 採用時期 | どのSprintで着手するか |
| 影響範囲 | 変更が及ぶモジュール・レイヤー |
| 推奨度 | 下記5段階 |

**推奨度ランク**

| ランク | 意味 |
|---|---|
| ★★★★★ | 必須 |
| ★★★★☆ | 強く推奨 |
| ★★★☆☆ | あれば良い |
| ★★☆☆☆ | 将来検討 |
| ★☆☆☆☆ | 好み |

---

## 7. レビューの実施タイミング

- 各Sprint終了時の振り返りとして実施する（原則）
- 上記に加え、気になった点があれば都度リクエストしてよい（随時レビューも可）

---

## 8. 最後に

レビューの目的は

**「より長く保守できるライフプランニングエンジンを作ること」**

である。

新しい技術を採用することや設計を複雑化すること自体は目的ではない。

YAGNIを尊重し、MVPの開発速度とのバランスを重視して提案してほしい。
