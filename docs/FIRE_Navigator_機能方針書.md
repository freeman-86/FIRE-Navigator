# FIRE Navigator 機能方針書
## ProjectionLab分析を踏まえた「採用／不採用／日本化／独自追加」の切り分け

前提: 本書はProjectionLab分析設計書をベースに、FIRE Navigator（Excelを入力・レポートUI、Pythonをシミュレーションエンジンとする自分専用ツール）の機能方針を定めるもの。ProjectionLabの再現ではなく、**「日本の税制・年金制度に最適化された、自分専用の意思決定支援ツール」**であることを判断軸とする。

判断基準は以下の4点：
1. **汎用性**: 国・制度に依存せず、FIRE計画の本質（資産形成・取り崩し・意思決定支援）に効く機能か
2. **必要性**: 自分専用ツールにとって過剰（マルチテナント・課金・広報系）でないか
3. **適合性**: 米国税制・年金制度に最適化されており、日本仕様への読み替えが必要か
4. **差別化余地**: ProjectionLabにない、日本仕様・自分専用だからこそ足せる機能か

---

## 1. 採用する機能（そのまま踏襲する設計思想）

制度に依存せず、FIRE計画ツールの本質的価値を担っている機能。ProjectionLabの設計思想として高く評価した部分（分析書8章）と直結する。

| 機能 | 採用理由 |
|---|---|
| **決定論的シミュレーション（基本モード）** | 単一仮定での将来予測は全ての土台。まずここが正確でなければ他の高度な機能は意味を持たない。 |
| **ネットワース推移グラフ（資産構成別積み上げ）** | 「今どうなっているか」を一目で把握できることはUXの核。Excel上でも再現価値が最も高い。 |
| **マイルストーンシステム（複数条件トリガー）** | 退職年齢・FI達成条件などを「イベント」として抽象化する設計は、日本仕様に変えても骨格は共通。条件分岐（例：市場悪化時の対応）を表現できる柔軟性はそのまま活かす。 |
| **キャッシュフロー優先順位（貯蓄→投資→返済等のルール化）** | 「余ったお金をどう配分するか」を毎年自動判定する仕組みは、自分の家計運用ルールをコード化する上で不可欠。 |
| **キャッシュフローの可視化（Sankey的な資金フロー図）** | 数字の羅列より「お金の流れ」を直感的に理解できる価値は制度に依存しない。分析書6章で評価した通り、意思決定の質を上げる。 |
| **モンテカルロ／ヒストリカルバックテスト** | 「決定論的な1本の予測線」だけでは不安が残る。過去データやランダム試行による成功確率の提示は、自分の意思決定の納得感を支える。 |
| **予測 vs 実績の分離・比較（Progress思想）** | 「計画」と「実績」を分けて記録し比較する設計は、長期運用ツールとして継続利用のモチベーションになる。自分専用だからこそ、実績データの蓄積価値は大きい。 |
| **シナリオ比較（Compare）** | 「早期退職 vs 継続勤務」「賃貸 vs 持ち家」等の意思決定は、複数シナリオを横に並べて比較できてこそ機能する。FIRE Navigatorの主目的そのもの。 |
| **感度分析（前提条件を動かして即座に再計算）** | 成長率・インフレ率を変えた時の反応をすぐ見られることは、Excelの「セルを変えたら再計算される」という体験と親和性が非常に高く、むしろExcel UIの強みとして活かせる。 |
| **取り崩し順序（Withdrawal Strategy）の設定** | 退職後にどの口座から取り崩すかは税負担に直結する重要な意思決定であり、制度が変わっても「順序をルール化する」という骨格は共通。 |
| **カスタムプロット** | 自分が見たい指標を自由に選んでグラフ化できる柔軟性は、自分専用ツールほど価値が高い。 |

## 2. 採用しない機能（意図的に外す）

SaaSとしての事業運営・マルチテナント対応・米国以外の各国対応など、「自分専用の日本向けツール」には過剰、または的外れな機能。

| 機能 | 不採用理由 |
|---|---|
| **Advisor Dashboard（Pro版・複数クライアント管理）** | FIRE Navigatorは自分専用であり、他者のクライアントを管理する必要がない。開発コストに対するリターンがゼロ。 |
| **サブスクリプション・課金・Pricing/Upgrade画面** | 販売しないツールに課金導線は不要。むしろ「自分の全データを自分で完全に所有する」という自己完結性こそが差別化価値。 |
| **フリーミアム段階開放設計（無料版のデータ非保存等）** | 顧客獲得導線の設計であり、自分専用ツールには無関係。全機能を最初から自分のために開放する。 |
| **Discord／コミュニティ機能・フィードバック導線** | 製品としてのユーザーコミュニティ運営は不要。GitHub Issuesなど開発者向けの管理で十分。 |
| **Plugin API（YNAB・Monarch Money等の外部連携）** | 米国の家計簿サービスとの連携であり、自分が使わないサービスとの連携コストは無駄。ただし「外部データを取り込んで実績を自動更新する」という思想自体は独自機能として日本のサービス向けに作り直す（4章参照）。 |
| **多国籍税制フル対応（米国以外にも英・豪・独・西など広く対応）** | 汎用SaaSとして各国ユーザーを取り込むための投資であり、自分専用ツールには日本の制度精度だけを追求すればよい。対応国を広げるコストをゼロにし、その分日本の制度精度に全振りする。 |
| **米国固有の税優遇制度そのもの（401k, Roth Conversion, QCD, DAF, 529など）の直接実装** | 制度そのものが日本に存在しない。ただし「税優遇口座を使い切る最適化」「変換戦略による生涯税負担最適化」という発想は日本版（iDeCo・NISA）に置き換えて採用する（3章参照）。 |
| **Estate機能の米国連邦・英国相続税モデル** | 対象制度が異なるため直接転用不可。日本の相続税制度に作り直す（3章参照）。 |

## 3. 日本仕様へ変更する機能（発想は継承・中身を作り直す）

ProjectionLabの「機能カテゴリ」は有用だが、中身が米国税制・年金制度に強く依存しているため、日本の制度に合わせて再設計する。

| 機能カテゴリ | ProjectionLab仕様 | 日本仕様への変更内容 | 変更理由 |
|---|---|---|---|
| **税優遇口座** | 401k, IRA, HSA, 529, Roth | **NISA（成長投資枠／つみたて投資枠）、iDeCo、企業型DC、財形貯蓄**に置き換え | 制度が根本的に異なる。非課税枠・拠出上限・受給時課税ルールをすべて日本の法令ベースで再実装する必要がある。 |
| **所得税シミュレーション** | 米国連邦税＋州税ブラケット | **所得税（累進課税）＋住民税（一律10%相当）＋社会保険料（健康保険・厚生年金・雇用保険）＋各種控除（給与所得控除・基礎控除・配偶者控除等）** | 日本は社会保険料の負担が可処分所得に与える影響が大きく、米国モデルをそのまま使うと誤差が大きすぎる。 |
| **Roth変換シミュレーション** | Roth IRAへの変換タイミング最適化 | **iDeCo受給方法（一時金 vs 年金 vs 併用）の税負担比較、退職所得控除の枠を意識した受取タイミング最適化** | 「税優遇の出口戦略を最適化する」という発想は同じだが、対象制度・控除の計算式を日本の退職所得課税ルールに合わせて作り直す。 |
| **公的年金（Withdrawal Strategyの一部）** | 米国Social Security（受給開始年齢の繰上げ/繰下げ） | **国民年金・厚生年金の繰上げ受給（最大60歳まで）／繰下げ受給（最大75歳まで）による受給額増減のシミュレーション、ねんきん定期便の見込額を入力データとして取り込み** | 日本のFIRE計画において公的年金の受給設計は資産寿命に直結する重要変数であり、米国のSocial Security計算式は使えない。 |
| **Estate（相続）** | 米国連邦遺産税、英国相続税 | **日本の相続税（基礎控除＝3,000万円＋600万円×法定相続人数）、生命保険金の非課税枠（500万円×法定相続人数）を踏まえた純遺産（Net Legacy）計算** | 相続税の計算式・控除体系が全く異なるため、制度をゼロから実装する必要がある。 |
| **通貨・地域設定** | 多通貨対応（USD, GBP等） | **円建て一本化**（海外資産を保有する場合のみ為替換算オプションを追加） | 自分専用ツールで多通貨対応の複雑性を持つ必要がない。ただし為替リスクの感応度分析は独自機能として別途追加（4章）。 |
| **モンテカルロ／バックテストの市場データ** | 米国市場中心の過去データセット | **日本株（TOPIX等）、全世界株、国内外債券の過去データセットに差し替え** | 自分の実際のポートフォリオ（全世界株中心など）の値動きに即したシミュレーション精度を優先する。 |
| **住宅資産（Real Assets）の税務処理** | 米国の住宅ローン控除・キャピタルゲイン控除 | **住宅ローン控除（住宅借入金等特別控除）、居住用財産の3,000万円特別控除等、日本の不動産税制に合わせる** | 不動産売却時の税務処理が日米で大きく異なるため。 |

## 4. 追加する独自機能（ProjectionLabにない、日本仕様・自分専用だからこそ作る機能）

分析書9章で挙げたProjectionLabの改善余地、および「Excel×Python」という役割分担・自分専用という前提を活かして、ProjectionLabにない付加価値を作る。

| 機能 | 内容 | 追加理由 |
|---|---|---|
| **Excel入力UIとの双方向連携** | Excelの名前付き範囲・シート構造とPythonエンジンを直結し、Excel上のセル変更が即座にPython側の再計算・レポート生成に反映される仕組み | ProjectionLabで指摘した「入力パネルとグラフを同時参照しづらい」という弱点を、使い慣れたExcelというUI資産で構造的に解決できる。分析書9章の弱点3の裏返し。 |
| **シナリオのGit管理（プラン変更履歴のバージョン管理）** | プラン設定ファイル（YAML/JSON等）をGitで管理し、いつ・どの前提を・なぜ変えたかを差分として追跡できる仕組み | ProjectionLabにはない発想。マルチクライアント管理は不要な代わりに、自分の意思決定の変遷そのものを資産として残せる。プロジェクト方針（GitHub前提・保守性重視）とも整合。 |
| **公的年金試算の自動取込** | ねんきん定期便・ねんきんネットの見込額データを入力し、繰上げ/繰下げ受給の比較を自動生成 | 日本のFIRE計画で公的年金の扱いは特に重要な変数であり、実データに基づく精度の高いシミュレーションを自分専用だからこそ丁寧に作り込める。 |
| **ふるさと納税シミュレーション** | 想定所得・控除に基づく年間上限額の算出と、返礼品込みの実質負担の可視化 | 日本の家計最適化に直結する身近な節税手段であり、ProjectionLabには存在しない日本固有の需要。 |
| **NISA/iDeCo枠の使い切り最適化ロジック** | 年間拠出余力に対し、NISA成長投資枠・つみたて投資枠・iDeCoへの配分を、税制メリットが最大になる順序で自動提案 | 米国のRoth変換最適化に相当する「出口戦略の最適化」を、日本の非課税制度の「入口（拠出）最適化」として独自に設計する。 |
| **家計簿サービス連携によるProgress自動更新** | マネーフォワードME等のエクスポートデータを取り込み、Current Finances／Progressを自動更新 | ProjectionLabのPlugin API（YNAB等）に相当する発想を、自分が実際に使っている日本のサービス向けに作り直す。手動更新の手間を減らし、実績記録の継続率を上げる。 |
| **為替リスク感応度分析** | 海外資産（全世界株等）を保有する場合の円換算後の資産推移を、為替レートシナリオ（円高/円安）別に表示 | 日本人が国際分散投資を行う際に固有の重要な変数だが、ProjectionLabの多通貨対応は「表示通貨の切替」に留まり、為替変動リスクそのものの感応度分析ではない。 |
| **シナリオ回帰テスト（意図しない計算結果変化の検知）** | Excel入力や税制パラメータを変更した際、既存シナリオの計算結果が意図せず変わっていないかを自動テストする仕組み | 商用ソフトウェアレベルの品質・保守性を目指す方針（プロジェクト方針）に基づく。ProjectionLabのようなSaaSは内部でこうしたテストを持っているはずだが外部からは見えない部分であり、FIRE Navigatorでは最初から設計に組み込む。 |
| **退職所得控除 最適受取シミュレーション** | 退職金・iDeCo一時金の受け取り年をずらした場合の退職所得控除の再計算（複数回受給時の重複排除ルールを含む）まで踏み込んだシミュレーション | 日本の退職所得課税は受給間隔によって控除額が変わる複雑な制度であり、正確にモデル化できれば大きな節税インパクトがある。自分専用だからこそ精度を追求する価値がある。 |

## まとめ：方針の一言表現

- **採用**：制度に依存しない「計画・可視化・比較・検証」の骨格はProjectionLabから継承する。
- **不採用**：事業運営・多国対応・米国固有制度は、自分専用ツールにとって無駄なコストなので切り捨てる。
- **日本化**：税・年金・相続・不動産まわりは、日本の制度に合わせてゼロから精緻に作り直す。ここがツールの信頼性を左右する核心部分。
- **独自追加**：Excel×Python構成と自分専用という前提を活かし、ProjectionLabにはない「Gitによる意思決定の履歴管理」「日本固有の節税シミュレーション」「回帰テストによる品質担保」を強みにする。
